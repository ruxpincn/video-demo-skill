# Successful Hotspot Tutorial Requirements Map

Use this map when recreating or extending the hotspot tutorial style.

| Requirement | Implementation method | Verification |
|---|---|---|
| Click actions must be obvious. | Animate the mouse moving to the target first, lower opacity while moving, then add a press animation. | Watch the frame before each click; cursor must visibly arrive before press. |
| Do not use red click dots. | Use an SVG mouse pointer and `.press` animation only. | No red dot or circular marker appears in extracted frames. |
| Every step must display completely for at least 1 second. | Put `settle` and `frame` actions with `duration >= 1000`. | Timeline playback does not rush directly into the next action. |
| Hotspot trend should not instantly pile up all materials. | First draw/blink the frame, then click, then zoom-open the next page. | The frame is visible before the click and transition. |
| First DeepTiming entry needs an animation, not a static box. | Use `navTour`: cursor passes nav items left-to-right; each title glows and scales in place; final DeepTiming item gets a click. | Titles enlarge near their original positions by about 1.5x. |
| Nav title enlargement must be precise. | Define nav boxes in source-image pixels and use a screenshot-backed `navPop` layer at the same center. | The enlarged title overlays the original text, not an offset duplicate. |
| Final DeepTiming stop needs a click. | Set final nav item `{ final: true, click: true }` and record a click event. | Cursor presses on DeepTiming before moving to the next step. |
| Page opened by click needs a zoom-open transition. | Use `zoomTransition` from the clicked box to the next screenshot. | The new page pops open from the click origin. |
| Remove dark/red residual icon after opening hotspot trend. | For sensitive transitions use a clean transition: hide overlays and set the destination image immediately before/inside zoom. | The new page has no leftover dark overlay or red icon from the prior page. |
| No white circular glow inside highlight boxes. | Remove radial glow pseudo-elements; keep only border, dashed blink, and selected-content brightening. | Inside the frame there is no white circle or flash. |
| Dashed frame should blink three times. | Use a frame animation with repeated opacity/border pulse, not a single persistent static border. | The frame pulses three times before settling. |
| Selected content should be brighter, not darker. | Use a dim overlay outside the region and a `focus-highlight` layer with the original screenshot inside the region. | Highlighted content remains readable and brighter than surroundings. |
| Third step: word cloud first, then freshness, then scroll down. | Timeline: frame word cloud 1s, zoom word cloud 2s, frame freshness 1s, zoom freshness 2s, scroll cue, scroll page transition down. | Key frames show word cloud before freshness, then down arrow and page motion. |
| Third step down arrow must be centered. | Compute arrow position from screen dimensions, not hotspot center: `screenW * 0.5`, `screenH * 0.5`. | Arrow appears in page center. |
| Up/down arrows should be bigger and emit light. | Increase `.scroll-arrow` dimensions, enlarge the arrow stem/head, add drop-shadow glow, and add a soft animated radial aura behind the arrow. Update JS centering offsets to match the new dimensions. | Preview the scroll cue; arrow is large, centered, and visibly luminous without covering critical content. |
| Large scroll arrows should be about two-thirds of the screen height when requested. | Set arrow dimensions dynamically from the active `.screen` height, such as `arrowH = screenH * 2 / 3`, `arrowW = arrowH * 0.72`, then center using the rendered arrow dimensions. | Runtime inspection shows arrow height variable is about 0.667 of screen height; visual bounding box may be slightly larger during scale animation. |
| Scroll arrows must move in their actual direction. | Use separate keyframes: down starts above center and moves below center; up starts below center and moves above center. | Runtime computed transform has positive Y during down motion and negative Y during up motion; screenshots show directional movement. |
| All tutorial materials must be sized for 4K, not left at review-page size. | In 4K `recording-export` styles, scale explanation labels, cursor, arrows, focus frames, scroll cue, zoom-preview borders, and nav pops together. Use dynamic measurements such as `offsetWidth/offsetHeight` for centered assets after resizing. | 4K extracted frames show labels, arrows, cursor, and frames are readable and visually intentional. |
| Overlay materials must not cover each other or drift away from the target. | Keep labels close to their highlighted target but outside the target bounds. Clear labels/frames before zoom-preview close-ups when the preview would overlap them. Do not keep arrows visible over explanation boxes. | Key frames show each active material has its own space; no zoom preview hides a black label, and labels are not stranded far from the selected frame. |
| Down arrow should trigger a downward camera/page transition. | After `scroll` action, run `scrollPageTransition` with `direction: "down"`. | The next screenshot enters with downward scroll motion. |
| Fourth step: industry rank first, question extension second, then scroll up. | Timeline: frame `industryRank`, frame `questionExtend`, then centered up arrow and `scrollPageTransition` direction up. | Extracted frames show that order. |
| Up arrow should trigger an upward camera/page transition. | Use `scrollPageTransition` with `direction: "up"`. | Previous page returns with upward motion. |
| Click on 算力 must be precise; do not magnify inaccurate crop. | Use a direct cursor click on the `powerWord` box, without frame or text-pop effect when the crop is uncertain. | Cursor presses on the intended word area; no imprecise enlargement appears. |
| Fifth-step click result must not flash. | Use instant image swap/no-transition inside `zoomTransition` around the 72% mark. | The modal opens smoothly without a one-frame flicker. |
| Anthropic click follows the same pattern. | Frame/cursor target precisely, click, then zoom-open to the detail modal. | Cursor click and modal zoom are both visible. |
| Explanation text should match the evidence-chain narrative. | Write labels around: market is trading what, hotspot trend, news/research/funds evidence chain,关注/利好/情绪强度. | Label copy is specific to the current step. |
| Remove "信谛听证据链查看流程" guide box. | Hide `.guide` globally or in `recording-export`; hide `.end-card` during recording. | `getComputedStyle(.guide).display` is `none`; no guide appears in video frames. |
| Black explanation box must be bigger and readable. | Style `.focus-label` with black translucent background, yellow title, white body, roughly 26px/19px baseline at 1080P. Use strong border, glow, padding, and shadow so it reads as a deliberate callout. | Extracted frames show readable, prominent labels. |
| Explanation box must not be too far or too small. | Place labels close to the active hotspot edge so the viewer sees one visual group. Use a larger full label for normal sections; use `.compact` only near large centered modals where margins are narrow. | Contact-sheet frames show the label adjacent to the highlighted region, not stranded in a far corner. |
| Explanation box must not cover the highlighted thing. | Place labels outside active hotspot bounds; for modal steps use side margins and compact max-width. | Inspect key frames for no overlap between label rect and active frame. |
| Do not export video while the user is still previewing layout tweaks. | For small visual changes, edit the webpage only and ask the user to refresh/preview. Export video only after the user confirms the page or explicitly asks for export. | Final response for tweak turns says no video was exported. |
| Hotspot word cloud and freshness labels should align above their frames. | Move these black labels upward enough to avoid covering the original dashed border while keeping them visually attached to the highlighted region. | Preview frames show labels aligned above the active boxes, not overlapping the border. |
| Hotspot question extension label should not be too high or too low. | Place the label between the previous section and the question-extension section, adjusting by visual preview rather than a fixed universal value. | Label is close enough to the question-extension area but does not cover the content below. |
| Final "三件事" conclusion callout. | Remove "结论页" from the title, use title "三件事", add a dedicated `.final` label style larger than normal labels, center it slightly above the modal center, and wait about 2 seconds on the clean final page before showing it. | Final page appears clean first, then the large centered black callout appears with title "三件事". |
| No residual effects after moving to the next step. | Centralize overlay cleanup and call it at step settle, zoom-preview end, scroll-cue start, zoom-transition start/end, and scroll-page-transition end. Clear dim, highlight, frame, label, mouse, zoom preview, nav pop, scroll cue/arrow, page zoom, and temporary image scroll classes. | Automated or manual audit shows no lingering `visible`, `active`, `show`, `press`, `moving`, `scroll-current`, or `scroll-next` classes after the next page/step starts. |
| Every webpage edit must be checked for residual effects before replying. | After any change to layout, copy, timing, frame, cursor, arrow, zoom, or transition behavior, run a browser preview/runtime audit on the changed step and the immediately following step. Inspect overlay DOM classes and/or screenshots for leftover dim layers, highlights, frames, labels, arrows, zoom previews, page zoom layers, nav pops, mouse states, and temporary scroll classes. | Final response should only say the page is ready after the residue audit passes, or explicitly say the audit could not be completed and why. |
| Export clarity must be at least 1080P. | Record Chrome viewport at 1920x1080 and encode H.264 with CRF around 20. | `ffprobe` reports width 1920 and height 1080. |
| Final delivery should match requested video frame rate, currently 50fps. | Export a correct-timing 4K master first, time-aligning captured frames to the page timeline if browser screenshot capture is slow. Then convert the finished master with `ffmpeg -vf "fps=50"` while copying or preserving audio. Avoid any method that changes duration. | `ffprobe` reports `avg_frame_rate` as `50/1`, video/audio duration remains close to the page timeline/event JSON, and audio stays present. |
| Higher frame rate must not make the demo feel rushed. | Increase timeline durations when needed, then verify the final video duration grew accordingly. Do not judge only by CSS or action durations; compare `pageElapsed` and `videoDuration` in the events JSON. | Events JSON has `scale` close to 1.0; final `ffprobe` duration matches the intended page timeline. |
| Video must not stretch. | The page constrains `.screen` to source aspect ratio; the stage background is black for letterboxing. | Screenshots look natural, with black bars if needed. |
| Sound effects should be in the video, not the webpage. | Record event times, synthesize click/zoom/scroll WAV after capture, then mux AAC into MP4. | `ffprobe` shows both video and audio streams. |
| Bad browser audio should be removed. | Do not use Web Audio in the page for final output; post-produce sound in export script. | No browser-generated sound is needed during capture. |
| Final answer should not claim success before checking. | Run `ffprobe`, extract contact sheet, and visually inspect key frames. | Final response can cite video path and verification result. |
